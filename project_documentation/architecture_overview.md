# Архитектура приложения


## 1. Цель документа

Определить **структуру приложения**, **ответственность модулей** и **правила взаимодействия** между ними. Документ служит основой для разработки и принятия архитектурных решений.

---

## 2. Архитектурные принципы

- **Модульность** — каждый модуль независим и отвечает за одну область.
- **Событийная модель** — модули общаются через события, а не прямые вызовы.
- **SOLID** — проектирование классов и модулей с минимальной связностью и максимальной расширяемостью.

---

## 3. Основные модули

- **Core** — EventBus, состояние приложения, логирование, конфигурация.
- [**UI**](/project_documentation/UI/UI_context.md) — визуализация данных, обработка пользовательских действий.
- **Database** — работа с локальной БД, загрузка/сохранение сущностей.
- **Email** — подключение к почтовым ящикам, загрузка и парсинг писем.
- **AI** — классификация, извлечение данных, генерация текстов.
- **Logistics** — статусы грузов, интеграции, обновления.
- **Documents** — шаблоны, генерация документов, экспорт.

Каждый модуль имеет **чёткий контракт** и **не знает деталей других модулей**.

---

## 4. Границы модулей

- Модуль не вызывает методы других модулей напрямую.
- Взаимодействие происходит только через **события** и **контракты**.
- Модуль может быть заменён без изменения остальных (OCP, DIP).

---

## 5. Модель взаимодействия

### 5.1 EventBus

Центральный механизм обмена сообщениями. Модуль может:

- **опубликовать событие**
- **подписаться на событие**

EventBus не знает, кто публикует и кто слушает.

### 5.2 События

Событие — это **структура данных**, описывающая факт, например:

- `EmailFetched`
- `EmailClassified`
- `CargoUpdated`
- `DocumentGenerated`

События не содержат логики.

### 5.3 Контракты модулей

Контракт включает:

- интерфейсы (например `IEmailClient`, `IDocumentGenerator`)
- список событий, которые модуль **генерирует**
- список событий, которые модуль **слушает**

Контракт — это “официальный язык общения” модуля с системой.

---

## 6. Потоки данных (пример)

1. Email → публикует `EmailFetched`
2. AI → слушает `EmailFetched`, публикует `EmailClassified`
3. Database → слушает `EmailClassified`, сохраняет данные
4. UI → слушает `EmailClassified`, обновляет интерфейс
5. Documents → слушает `EmailClassified`, предлагает шаблоны

Модули не знают друг друга — только события.

---

## 7. Расширяемость

Чтобы добавить новый модуль:

1. Создать папку.
2. Определить контракт (интерфейсы + события).
3. Зарегистрировать подписки в Core.
4. Реализовать модуль.

Существующие модули менять не требуется.
